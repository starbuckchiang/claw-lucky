Architecture Review
⭐⭐⭐⭐⭐ Provider Boundary

我最先看的就是這一句：

Gemini SDK 僅出現在 Provider Adapter 邊界。

這代表你成功維持了：

Client
    ↓
API
    ↓
Orchestrator
    ↓
Generation Service
    ↓
Provider Adapter
    ↓
Gemini SDK

如果 SDK 出現在 Generation Service，我會直接要求重構。

這部分完全符合 ADR-001 的精神。

⭐⭐⭐⭐⭐ Dependency Injection

Review 提到：

新增 GeminiProviderAdapter，由 provider factory 注入，不修改 Generation Service。

這表示未來要加入：

OpenAI Images
Azure OpenAI
Replicate

理論上都不需要修改 Business Layer。

這就是當初做 Provider Adapter 的價值。

⭐⭐⭐⭐⭐ Configuration

我特別看了這段：

MODEL
TIMEOUT
MAX_RETRY
IMAGE_SIZE
SAFETY_LEVEL
API KEY

全部由環境變數提供，沒有硬編碼。

這是正式專案必備條件。

⭐⭐⭐⭐⭐ Response Mapping

Review 明確指出 Gemini 回應已映射為統一的 NormalizedProviderResponse，包含：

provider
model
finishReason
image
usage
durationMs
rawResponse（僅 Debug）

這讓 Business Layer 不需要理解 Gemini 的原始格式。

⭐⭐⭐⭐⭐ Error Mapping

這也是我很在意的地方。

目前已統一為：

PROVIDER_TIMEOUT
PROVIDER_RATE_LIMIT
PROVIDER_AUTH_FAILED
PROVIDER_BAD_REQUEST
PROVIDER_UNAVAILABLE
PROVIDER_UNKNOWN

代表未來換 Provider 時，上層程式幾乎不用改。

⭐⭐⭐⭐⭐ Observability

Review 確認：

Structured Log
Correlation ID
不記錄 Prompt 全文
不記錄 Secret

完全符合 ADR-008。

Testing

目前測試涵蓋：

Success
Timeout
Rate Limit
Auth Failure
Retry Success
Retry Failure
Response Mapping
Error Mapping
Correlation Propagation

而且全部使用 Mock SDK，沒有直接呼叫真實 Gemini。

這是正確的測試策略。

我提出兩個 Minor 建議（不阻擋）
① Provider Capability Registry

目前只有 Gemini。

建議之後建立：

provider-capabilities.js

例如：

{
  gemini: {
    supportsImage: true,
    supportsStreaming: false,
    supportsReferenceImage: true,
    supportsFaceSwap: false
  }
}

未來加入不同 Provider 時，就不用在各處寫判斷。

② Provider Health Check

之後可以加入：

provider-health.js

提供：

Provider 是否可用
最近錯誤率
平均回應時間

這會對後續營運很有幫助，但不屬於 P2-AI-01 範圍。

建議新增文件

我建議建立：

docs/testing/provider-manual-test.md

內容包括：

如何設定 .env
如何切換 USE_REAL_PROVIDER=true
如何執行一次真實生圖
如何確認回傳圖片
如何查看 Correlation ID
常見錯誤與排查方式

這會比把操作散落在 README 更容易維護。
